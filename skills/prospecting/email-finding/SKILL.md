A single batch of\nbounces can require months of domain warmup to recover from.\n\nThis skill walks through email discovery using a multi-provider waterfall —\nstarting with the highest-accuracy source first, falling back only when that\nsource returns empty. Every email found must be verified before entering any\noutbound sequence.\n\n## When to Use\n\n- \"Find the email for John Smith at Acme Corp\"\n- \"Get me verified emails for this list of contacts\"\n- \"What's the email pattern for company.com?\"\n- \"Look up Jane's work email\"\n- \"Enrich this CSV with email addresses\"\n\nDo NOT use for:\n- Phone number finding — that's a separate waterfall\n- Full company research — use lead-enrichment first\n- Already have emails — skip to contact-verification\n\n## Authoritative Foundations\n\nEmail finding follows waterfall enrichment principles from GTMLens (5-layer\nwaterfall: Apollo → ZoomInfo → PDL → Claygent → Claude normalization) and\nZiellab's 3-waterfall architecture (separate company, email, and phone\nwaterfalls run independently).\n\nThe core insight: no single provider covers more than 60-75% of B2B contacts.\nChaining 3-5 providers in sequence routinely pushes coverage to 85-92%.\n\n## Prerequisites\n\n- First name + last name + company domain (minimum)\n- LinkedIn URL significantly improves match rates (15-25 percentage points)\n- If using a provider like Apollo, ZoomInfo, or Clay: valid API key or active subscription\n\n## Step-by-Step Process\n\n### Phase 1: Intake\n\nAsk the user for:\n1. How many contacts need emails? (single lookup vs batch)\n2. What data do you have for each? (name, company, LinkedIn URL)\n3. What providers do you have access to? (Apollo, ZoomInfo, Clay, etc.)\n4. What's the budget per contact? (emerging vs enterprise providers)\n\n### Phase 2: Provider Selection\n\nOrder providers by cost-per-hit for the user's specific ICP segment:\n\n| Provider | Best For | Match Rate Est. | Cost Profile |\n|---|---|---|---|\n| LeadMagic Email Finder | US B2B, all segments | 60-75% | Pay-per-result |\n| Apollo | SMB, mid-market, US | 60-75% | Subscription-based |\n| ZoomInfo | Enterprise, Fortune 5000 | 30-40% incremental | High, annual contract |\n| People Data Labs | Technical roles, EU, early-stage | 10-18% incremental | Moderate, API credits |\n| Hunter.io | Domain-pattern matching | 15-25% | Low, per-request |\n| Dropcontact | GDPR-compliant, EU | 10-20% | Moderate |\n| Claygent (Clay) | Founders, execs, web-visible | 5-15% incremental | Credits per search |\n\nBuild the waterfall from cheapest + highest-coverage first, falling back to more\nexpensive providers only when earlier steps miss.\n\n### Phase 3: Run the Waterfall\n\nFor each contact:\n\n1. **Primary provider** — Run the highest-accuracy, lowest-cost provider first.\n   For most US B2B: LeadMagic Email Finder (returns verified results) or Apollo.\n\n2. **Fallback 1** — If primary returns empty, run secondary provider.\n   ZoomInfo for enterprise, Hunter for domain-based lookup.\n\n3. **Fallback 2** — If still empty, run tertiary.\n   People Data Labs or Dropcontact depending on region.\n\n4. **AI-assisted (Claygent)** — For remaining 10-15%, use Claygent with an\n   explicit prompt: \"Find the work email for [name] at [company]. Return the\n   email address AND the source URL. Do NOT guess or construct emails from\n   patterns. If no verified source found, return empty.\"\n\n5. **Normalization** — Consolidate results. Use COALESCE logic: prefer results\n   from earlier waterfall steps. Prefer verified over pattern-matched.\n\n### Phase 4: Verification\n\nEvery found email must be verified before use. Run verification as a separate\nstep — never skip it. See contact-verification skill.\n\nAcceptable verification results: valid only. Handle risky/catch-all results\nseparately (lower volume, extra monitoring). Discard invalid results.\n\n## Output Format\n\nFor a single lookup:\n```\nContact: John Smith, Acme Corp\nEmail: john.smith@acme.com\nSource: LeadMagic Email Finder\nStatus: Verified — valid\nConfidence: High\n```\n\nFor batch enrichment, produce a CSV with columns:\n`email`, `email_source`, `email_confidence`, `verification_status`\n\n## Quality Check\n\n- [ ] Every email found has a source attribution\n- [ ] Every email has been run through verification\n- [ ] Bounce rate target: under 2% across the batch\n- [ ] No pattern-guessed emails without source confirmation\n- [ ] Catch-all domains flagged separately\n\n## Common Pitfalls\n\n1. **Skipping verification.** Unverified emails cause bounces. Bounces damage\n   sender reputation. Recovery takes weeks. Always verify after finding.\n\n2. **Single-provider dependency.** One provider covers 60-75% max. Running\n   only Apollo leaves 25-40% of contacts unreachable. Always waterfall.\n\n3. **Using pattern-guessed emails.** Constructing firstname.lastname@company.com\n   without confirmation creates 40-60% bounce rates. Never guess. Verify.\n\n4. **Running waterfall in wrong order.** An expensive provider first burns\n   budget on contacts a cheaper provider would have found. Sort by cost-per-hit.\n\n5. **Not normalizing company names.** \"Acme Inc\" vs \"Acme, Inc.\" vs \"Acme\n   Corporation\" create duplicate lookups. Match on domain, not name.\n\n6. **No LinkedIn URLs in input.** Adding LinkedIn URLs improves match rates\n   by 15-25 percentage points across all providers. Run a LinkedIn URL finder\n   column before email finding when URLs are missing.\n\n## Related Skills\n\n- **lead-finding**: Finds the contacts this skill enriches with email addresses\n- **lead-enrichment**: Full enrichment beyond email (firmographics, technographics)\n- **contact-verification**: Run after this skill — validates found emails before sending\n- **waterfall-enrichment**: Complete multi-field waterfall architecture\n
